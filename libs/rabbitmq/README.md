# @omni-bus/rabbitmq

RabbitMQ transport for [omni-bus](../../README.md) over [`amqplib`](https://github.com/amqp-node/amqplib). Supports topic-exchange fan-out for events and AMQP direct reply-to RPC for commands.

## Install

```bash
npm install @omni-bus/rabbitmq @omni-bus/core amqplib
```

`amqplib` is declared as a peer dependency.

## Quick start

```ts
import { OmniBus } from '@omni-bus/core';
import { RabbitMQTransport } from '@omni-bus/rabbitmq';
import { ClassTransformerSerializer } from '@omni-bus/class-transformer';

const transport = RabbitMQTransport.create({
  url: 'amqp://localhost',
  serializer: new ClassTransformerSerializer(),
  // Optional:
  exchangeName: 'my-app.exchange',  // default: 'omni-bus.exchange'
  queueName: 'my-app.consumer',     // default: auto-generated exclusive queue per instance
  rpcTimeoutMs: 30_000,             // default: 30s
  prefetch: 32,                     // default: 32 — per-consumer prefetch cap
});

const bus = OmniBus.create({
  transports: { rabbitmq: transport },
  defaults: { transport: 'rabbitmq' },
});

await bus.start();
```

## Topology

On `start()`, the transport:

1. Opens a connection + channel against the broker.
2. Asserts a **topic** exchange (`durable: true`).
3. Asserts a queue (exclusive auto-generated by default; explicit + non-exclusive if `queueName` is supplied).
4. Binds the queue to the exchange with two routing patterns: `event.*` and `command.*`.
5. Consumes from the bound queue (regular delivery) and from the `amq.rabbitmq.reply-to` pseudo-queue (for RPC replies, `noAck: true`).

| Routing key | Used for |
|---|---|
| `event.<messageType>` | Event fan-out. |
| `command.<messageType>` | Command delivery. |
| `amq.rabbitmq.reply-to` | RPC replies (AMQP's direct reply-to). |

## RPC correlation

`send(envelope)`:

1. Publishes to the exchange with `routingKey = command.<messageType>`, `replyTo = amq.rabbitmq.reply-to`, `correlationId = envelope.messageId`.
2. Stores `{ resolve, reject, timer }` in a `Map<correlationId, ...>` keyed by `envelope.messageId`.
3. Awaits a reply on `amq.rabbitmq.reply-to`, matched by `correlationId`.

`stop()` rejects every pending entry with `"RabbitMQTransport stopped while an RPC was pending."`.

## Wiring modes

The transport's `start({ inbound, replyListener })` decides which AMQP wiring is set up after the connection + channel + topic-exchange assertion (always done so outbound `publish` works):

| `inbound` | `replyListener` | What's opened |
|---|---|---|
| `false` | `false` | Connection + channel + exchange only. No queue assertion, no `consume`. Pure fanout publisher. |
| `false` | `true` | + `consume('amq.rabbitmq.reply-to', ..., { noAck: true })`. Publisher that can await RPC replies. |
| `true` | `false` | + `assertQueue` + `bindQueue('event.*' / 'command.*')` + `consume` on the bound queue. Consumer that never awaits RPC replies. |
| `true` | `true` | Full mode: bound consumer + reply-to consumer. The default. |

The bus derives these flags from the handler registry + routes; you usually don't pass them explicitly.

## Acks and prefetch

Inbound messages are acked **after** the bus's inbound dispatch returns (success or handler error). Deserialization failures are NACKed with `requeue: false` so malformed payloads don't redeliver forever. Each consumer is initialized with a `prefetch` cap (default 32) to bound unacked-in-flight on the broker.

- Successful dispatch → `ack`.
- Event handler throws → reported via the bus's `onError` hook, message is still `ack`ed (at-most-once semantics for events on this transport).
- Command handler throws → fault envelope sent back via `replyTo`, then `ack`. Caller's `bus.send()` rejects with the rehydrated error.
- Bad payload (deserialize fails) → `nack(allUpTo=false, requeue=false)`.

## Caveats

- **Auto-generated queue is exclusive.** With the default (no `queueName` supplied), each bus instance gets its own auto-named exclusive queue, deleted when the connection closes. This is correct for fan-out semantics. For a competing-consumers pattern across instances, supply an explicit `queueName` and the queue becomes shared and durable.
- **No automatic reconnect.** If the broker connection drops, `start()`/`stop()` must be re-issued. Reconnect logic is a v1.5 concern.
- **DLQ wiring** (dead-letter exchange) is not built in. It can be added by passing custom queue arguments via a future option — currently you'd configure DLQ at the broker level.
